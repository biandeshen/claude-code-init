# Agent 行为规则

> 版本：v1.0.0 | 更新：2026-04-21
> 本文档描述 Agent 执行任务时的行为规则，适用于任何 Agent 开发项目。

---

## 一、假设验证优先

### 1.1 核心原则

修改代码前，**必须**在对话中输出：
```
[假设] 我认为 [描述]，因为 [依据]
```

### 1.2 执行步骤

1. 输出假设到对话
2. 用 `Write`/`Edit` 将假设写入 `Plan.md`
3. 验证假设
4. 若假设被否定，**立即停止**，禁止原路径修补

### 1.3 示例

```
[假设] 我认为问题的原因是变量类型不匹配，因为错误信息显示 "TypeError: unsupported operand"
[计划] 修改 agent_loop.py 第 85 行，将参数类型从 str 改为 dict
```

## 二、错误即停（2 次熔断）

### 2.1 "同一操作"判定

满足以下任一条件即视为同一操作：

| 条件 | 说明 |
|------|------|
| 相同工具 + 相同参数 | 执行完全相同的命令 |
| 同文件同区块 + 相同性质修改 | 修改同一区域的问题 |
| 同目标 + 本质相同方案 | 解决同一个问题的不同尝试 |

### 2.2 熔断规则

| 失败次数 | 动作 | 使用的工具 |
|----------|------|------------|
| **1次** | 记录错误到 `Plan.md`，输出分析报告 | `Edit` 写日志 |
| **2次** | **立即停止**，请求人工介入 | `SendMessage` 求助 |

> 例外：外部环境变化导致失败（如网络中断），不计入连续次数。

### 2.3 示例流程

```
第1次失败：
  - [错误] 尝试 import yyy 失败
  - [动作] 记录到 Plan.md，输出分析

第2次失败：
  - [错误] 再次尝试 import yyy 失败
  - [动作] 立即停止，向用户报告
```

## 三、变更即记录

### 3.1 Plan.md 日志格式

每次修改后，用 `Edit` 在 `Plan.md` 追加：

```
HH:MM | [动作] | [结果] | [下一步]
```

### 3.2 示例

```
09:15 | 修改 event_bus.py | 成功 | 继续测试 publish 方法
09:20 | 测试 publish | 失败：事件未触发 | 检查订阅逻辑
09:25 | 检查订阅逻辑 | 发现问题：topic 不匹配 | 修复中
```

### 3.3 超时提醒

5 分钟无日志更新，用 `SendMessage` 主动询问用户。

## 四、主动升级

### 4.1 触发条件

以下情况立即暂停，请求确认：

| 触发条件 | 动作 |
|----------|------|
| 提交含 `!important`/`hack`/`fix` | `Bash: git log -p` 检查 |
| 影响超过 3 个组件 | `Grep` 检查引用 |
| 不可逆操作（数据库迁移、文件删除） | `SendMessage` 请求确认 |
| 假设被否定 | 立即停止，重新分析 |

### 4.2 可不等失败

主动升级不需要等待失败发生，发现风险即可暂停。

## 五、禁止规则（硬性拦截）

| 禁止项 | 说明 |
|--------|------|
| ❌ 假设未验证就 `Edit`/`Write` | 必须先输出假设 |
| ❌ 同一方案连续尝试 ≥3 次 | 超过熔断限制 |
| ❌ 忽略 `pytest` / `ruff` 错误继续执行 | 必须修复错误 |
| ❌ 未更新 `Plan.md` 连续修改多个文件 | 必须记录变更 |
| ❌ 未确认需求就写代码 | 必须先理解需求 |

## 六、工作流模式

### 6.1 代码审查工作流

```
Read(目标文件) → Bash(git log -p) → Grep(检查引用) → Task(分析报告)
```

### 6.2 Bug 修复工作流

```
Read(错误上下文) → Grep(定位问题) → [假设] → Bash(pytest) → Edit(修复) → Bash(验证)
```

### 6.3 重构工作流

```
Task(分解任务) → Read(理解结构) → Grep(检查依赖) → [假设] → Edit/Write(修改) → Bash(测试)
```
