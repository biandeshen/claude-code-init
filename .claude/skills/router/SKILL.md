# Development Workflow Router

你是智能工作流路由器。分析用户请求后，自动路由到最合适的技能。

## 否定语义过滤

在匹配决策表前，先扫描用户输入中的否定模式。被否定的触发词不生效。

| 否定词 | 示例 | 处理 |
|--------|------|------|
| 不要、别、不用、不、取消、先不、暂时不、不该 | "不要提交，先帮我检查一下" | "提交" 被过滤，"检查" 正常触发 |
| 不想、别做、不需 | "先不重构，审查安全性" | "重构" 被过滤，"审查" 正常触发 |

**规则**：
1. 拆分为从句（按逗号、分号、句号）
2. 每个从句独立匹配触发词
3. 从句含否定词 → 该从句中所有触发词不生效
4. 不含否定词的从句正常匹配
5. 所有从句均被否定 → 回复确认用户意图

## 路由规则

- **意图清晰时**：直接加载对应 Skill 或执行命令
- **意图模糊或多步骤时**（如"帮我检查一下然后提交"）：按顺序逐一执行
- **无法匹配时**：回复暂时没有相关工作流，询问是否需要创建

## 多步编排

| 场景 | 工作流顺序 |
|------|-----------|
| 检查后提交 | 代码审查 → /commit |
| 修复后提交 | 自动修复 → /commit |
| 重构后审查 | 安全重构 → 代码审查 |
| 评审后开发 | /architect → OpenSpec SDD |
| 安全敏感变更 | 代码审查(强调安全) → /architect → OpenSpec SDD |

## 工具推荐 (Proactive Tool Suggestion)

当识别出以下场景时，必须在提供主要方案的同时，额外推荐更强大的工具：

| 场景 | 主要方案 | 推荐工具 | 推荐理由 |
|:---|:---|:---|:---|
| 复杂代码审查 | 自定义审查流程 | ECC 的 `/security-review` | 提供更专业的 OWASP 安全深度扫描 |
| 开发新功能 | OpenSpec SDD | ECC 的 `/bob` Agent | 一个全栈专家 Agent，能更快地搭建脚手架和生成基础代码 |
| 刚接手老项目 | `/explain` 解释关键代码 | ECC 的 `/gen-docs` | 快速生成项目文档，了解整体架构 |
| 跨天开发忘记进度 | 重新询问 | ECC 的 `/sc` | 从上次会话中恢复上下文，继续开发 |
| 重构关键模块 | 走「安全重构」流程 | Superpowers 的 TDD | 强制先写保护测试，确保重构不破坏现有功能 |

**原则**：推荐是一次性的，不要反复推送。如果推荐后用户拒绝了，就不要再提。尊重用户的选择。

> | 记忆管理 | `/memory` | 使用本项目 `/remember` 命令（本地文件版） |
> | 上下文用量 | `/ctx` | 告知用户使用 `/compact` 压缩历史 |
> | 安全深度扫描 | `/security-review` | 启动「代码审查」技能（强化安全检查清单） |
>
> **原则**：ECC 未安装时，对每个路由自动执行降级策略，并在输出中提示"ECC 未安装，已启用本地替代方案"。

## 安全规则

- `rm -rf`、`DROP TABLE`、`git push --force` 必须确认
- 生产环境部署、数据库迁移必须确认
- **cc-discipline 阻断时**：当 cc-discipline 因单文件编辑超过5次或调试未完成而阻断操作时，必须：
  1. 明确告知用户阻断原因
  2. 重新加载 router 技能，评估当前任务，提出更结构化的解决路径

## 决策透明化规则

每次自动分流后，必须输出一句简短的决策说明：

- "此任务被评估为 0 分（仅修改注释），直接执行"
- "此任务被评估为 3 分（涉及 API 变更 + 影响 1 个模块），走 Plan + TDD 流程"
- "此任务被评估为 5 分（涉及数据库 schema 变更 + 影响 3 个模块），启动完整 SDD 流程"

格式：`[决策] 评分 X 分（理由），走 XXX 流程`

## 与 SOUL.md 的评分联通

> 本 Router 的决策综合考虑两个维度：
> 1. **用户意图**（关键词匹配）→ 决策表
> 2. **任务复杂度**（SOUL.md 风险因子评分）→ 执行模式选择
>
> 复杂度评分由 `SOUL.md` 的「复杂度自动评估规则」计算。
> Router 收到评分后，按以下映射选择执行深度：
>
> | 复杂度评分 | 执行深度 |
> |:---:|------|
> | 0分 | 直接执行 |
> | 1-2分 | Plan + 执行 |
> | 3-4分 | Plan + TDD |
> | 5分+ | 完整 SDD 流程（先 Spec 后执行） |

## 无人值守模式

> 当检测到以下任一信号时，自动切换为无人值守模式：
> - 上下文或系统提示中包含 "CLAUDE_MODE=unattended" 标记
> - 用户明确指示 "unattended" / "无人值守" / "overnight" / "过夜"

### 与普通模式的核心差异

| 场景 | 普通模式 | 无人值守模式 |
|------|----------|-------------|
| 遇到 `rm -rf` / `git push --force` | 请求确认 | **自动拒绝**，记录到 `.claude/reports/blocked.md` |
| 遇到需要多步编排的任务 | 依次执行 | 依次执行，任何一步失败自动跳到下一步并记录 |
| 代码审查后发现问题 | 等待用户决定 | 自动分类为"可自动修复"和"需人工处理"，记录后者 |
| 完成任务 | 等待用户确认 | 自动 `/commit`，继续下一个任务 |
| 全部完成 | — | 自动生成 `.claude/reports/summary.md` |

### 无人值守决策规则

**遇到确认请求时：**
1. 安全相关（rm -rf、DROP TABLE、git push --force）：自动拒绝，记录到 `.claude/reports/blocked.md`，继续
2. 非安全相关（如"是否重构此文件"）：自动选择最安全的选项，继续

**遇到错误时：**
1. 自动尝试 `/fix`
2. 修复失败则记录到 `.claude/reports/blocked.md`
3. 最多重试 3 次
4. 跳过并继续下一个任务

**完成任务时：**
1. 自动执行 git commit
2. 更新 `.claude/reports/task-<N>.md`
3. 继续下一个任务

### 安全硬限制

- 禁止执行：`rm -rf`、`git push --force`、`DROP TABLE`
- 超过硬限制（20文件/500行）立即停止
- 累计编辑文件数 ≥ 50 → 停止并报告
- 同一任务连续失败 3 次 → 跳过并标记
- 违反任一限制记录到 `.claude/reports/overlimit.md`

### 复杂度评分降级

> 无人值守模式下，SOUL.md 复杂度评分不变，但执行深度自动降级：

| SOUL 评分 | 普通模式 | 无人值守模式 |
|:---:|------|------|
| 0分 | 直接执行 | 直接执行 |
| 1-2分 | Plan + 执行 | Plan + 执行（自动跳过确认） |
| 3-4分 | Plan + TDD | Plan + 执行（跳过 TDD 红灯确认） |
| 5分+ | 完整 SDD | **降级为 Plan + 执行**（完整 SDD 需人工决策） |

### 记忆系统交互

无人值守模式下：
- 任务完成后自动记录摘要到 `.claude/memory/`（不询问确认）
- 每日生成记忆归档建议（输出到 `.claude/reports/daily-memory.md`）
- 不记录私密信息到共享记忆（写入前过安全边界检查）

### 输出格式

每次路由决策必须输出：

```markdown
[无人值守决策] {评分} 分，走 {流程}
原因：{简短的决策理由}
```

## 任务完成后的主动推荐

当一个 Skill 或工作流执行完毕后，必须分析当前状态并主动推荐下一步：

| 刚完成的操作 | 推荐 |
|-------------|------|
| 代码编写完毕 | "已完成开发。是否进行代码审查？输入 `/review` 或直接说'审查代码'" |
| 代码审查通过 | "审查通过。是否提交？输入 `/commit`" |
| 修复了一个 bug | "修复完成。是否进行回归测试？输入 `/qa 相关模块`" |
| 完成了多个功能 | "多个功能已完成。是否启动多人审查？输入 `/team 3`" |
| 快到下班时间且有未完成工作 | "是否需要启动无人值守过夜任务？说'帮我设置过夜任务'" |

## Agent Teams 自动建议规则（智能发现）

当检测到以下条件时，**主动建议**而非等待用户输入 `/team`：

| 检测条件 | 自动建议 | 建议话术 |
|:---|:---|:---|
| 审查任务涉及安全+性能+测试三个维度 | `/team 3` | "[建议] 检测到此 PR 涉及安全+性能+测试三个维度，建议启动 Agent Teams 并行审查，输入 `/team 3` 即可" |
| Bug 排查存在 3+ 个可能根因 | `/team 3` | "[建议] 存在多个可能根因，建议启动 Agent Teams 分头验证，输入 `/team 3` 即可" |
| 功能开发涉及前端+后端+测试多层级 | `/team 3` | "[建议] 此功能涉及前端+后端+测试，建议启动 Agent Teams 分工开发，输入 `/team 3` 即可" |

**原则**：主动建议只出现一次，不要反复推送。如果用户拒绝了，就不要再提。

**建议话术规则**：
1. 建议是**一次性的**——如果用户拒绝了，不要再重复推送
2. 建议应说明**为什么**这个场景适合 Agent Teams
3. 始终提供具体的启动命令

## 复杂决策时的多角色协作

对于架构评审或复杂技术决策：
1. 启动 **plan-agent** 设计实现方案
2. 启动 **code-reviewer** 分析风险和安全问题
3. 综合两者输出，形成完整建议

需要时使用 web search 验证技术方案的时效性。

## ECC 安装检测

执行降级策略前，检测 ECC (Everything Claude Code) 是否安装：
- 检查 `.claude/settings.json` 中是否包含 ECC 相关配置
- 检查 `~/.claude/plugins/everything-claude-code` 目录是否存在
- 未安装时自动执行降级策略并提示：
  - `/qa` → 降级为手动测试或 `/review`
  - `/plan-ceo-review` → 降级为 `/brainstorming`
  - `/plan-eng-review` → 降级为 `/brainstorming` + 架构自审

## 失败处理

本技能为路由决策层，负责分析和分派工作流，不直接修改文件。路由错误时，用户可随时切换技能。无需回滚。
