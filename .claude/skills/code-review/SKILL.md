# 代码审查协议

## 核心原则

作为审查者，你的任务是找出代码中真实存在的潜在风险，而不是进行代码风格辩论。

## 审查流程

### 1. 理解变更意图
首先，搞清楚这些代码改动是为了解决什么问题。
- 查看 git diff
- 对照 PLAN.md（如有）

### 2. 安全第一
严格检查所有用户输入、数据库查询、文件操作和外部API调用。

| 检查项 | 常见问题 |
|--------|----------|
| SQL注入 | 直接拼接SQL字符串 |
| XSS | 未转义的用户输入直接输出 |
| 敏感信息 | 密钥、token、日志泄露 |
| 权限认证 | 未校验用户权限就放行 |
| 依赖安全 | 使用有漏洞的第三方库 |

### 3. 逻辑与性能检查
- 是否有空指针/未定义变量的风险？
- 是否有N+1查询或性能瓶颈？
- 错误处理是否完善且安全？

### 4. 规范与架构一致性
- 是否遵守了项目 `CLAUDE.md` 中的约定？
- 是否引入了不必要的耦合？
- 函数是否过长（>50行）？

### 5. 测试覆盖
- 核心逻辑是否有测试？
- 边界情况是否覆盖？

## 输出格式

请将所有发现分类为：`致命(CRITICAL)`、`高危(HIGH)`、`中危(MEDIUM)`、`低危(LOW)`。

```markdown
### 审查摘要
- 变更文件数：X
- 发现问题数：X（致命/X 高危/X 中危/X 低危）

### 问题列表

| 严重性 | 文件 | 行号 | 问题描述 | 修复建议 |
| :--- | :--- | :--- | :--- | :--- |
| 🔴 致命 | ... | ... | ... | ... |
| 🟠 高危 | ... | ... | ... | ... |
| 🟡 中危 | ... | ... | ... | ... |
| 🟢 低危 | ... | ... | ... | ... |

### 总体评价
- ✅ **通过**：无致命或高危问题
- ⚠️ **建议修改**：有中低级问题，建议处理
- ❌ **需修复**：有致命或高危问题，必须处理
```

## 审查规则

- 发现致命问题时，必须阻止提交
- 审查过程不修改代码，只报告问题
- 使用多角色分析复杂变更
