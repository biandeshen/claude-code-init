# 需求头脑风暴与方案设计

## 核心原则
- 在完全理解需求之前，禁止写任何代码
- 先用一问一答的方式消除歧义
- 输出结构化方案，等待用户确认后再执行

## 执行流程

### 第一步：需求复述
1. 用你自己的话复述用户的需求
2. 明确目标、范围、验收标准
3. 如果不确定，主动提问澄清

### 第二步：方案输出
按以下结构输出方案：
1. **功能需求清单**：列出所有需要实现的功能点
2. **非功能性需求**：性能、安全、可维护性要求
3. **技术方案**：核心模块设计、数据流、接口定义
4. **风险点**：已识别的潜在问题和应对方案
5. **替代方案**：简要说明 1-2 个备选方案及其优劣

### 第三步：等待确认
- 输出完整方案后，明确询问用户是否同意
- 用户确认后再进入编码阶段
- 如果用户有修改意见，调整方案后重新确认

### 第四步：制定执行计划
- 将方案拆解为可独立执行的原子任务
- 按优先级排序
- 按 SOUL.md 复杂度评估规则为每个任务评分，触发对应的执行模式（由 Router 自动选择，本步骤不自行路由）
- 确认后推荐使用 `/architect` 进入技术架构设计阶段

## 失败处理

本技能为咨询类操作，不修改任何文件。

- **需求不明确时**：最多进行 3 轮澄清提问
- **3 轮后仍未明确**：输出当前已收集的信息摘要，标注未确认项，
  推荐用户使用 `/brainstorming` 或 `/plan-ceo-review` 进行更结构化的需求分析
- **用户明确表示不确定**：输出框架性建议（如"建议先用 MVP 验证核心假设"），然后退出
- 本技能无需回滚，不修改任何文件。
