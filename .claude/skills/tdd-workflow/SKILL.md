# TDD 工作流

## 核心原则
- 永远先写测试，再看测试失败，最后写实现代码
- 每一步必须通过验证后才能进入下一步
- 连续失败 3 次必须停下来讨论架构问题

## 执行流程

### 第一步：红（写失败的测试）
1. 分析目标功能，确定测试用例
2. 编写最小化的测试代码
3. 运行测试，确认测试**失败**（红色）
4. 如果测试不是因为正确原因失败（比如语法错误），修正测试

### 第二步：绿（写最简单的实现）
1. 编写刚好能让测试通过的最简代码
2. 不要提前做任何优化、抽象或额外功能
3. 运行测试，确认**通过**（绿色）

### 第三步：重构
1. 在测试全部通过的前提下优化代码结构
2. 消除重复、改善命名、提取公共逻辑
3. 每次重构后立即运行全部测试
4. 保持测试持续绿灯

### 第四步：提交
- 全部通过后执行 `/commit`

## 注意
- 禁止跳过测试直接写实现
- 禁止在测试失败时继续写新功能
- 如果是修复 Bug，必须先写复现测试
