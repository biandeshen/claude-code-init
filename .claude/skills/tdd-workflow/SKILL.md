# TDD 工作流

## 核心原则
- 永远先写测试，再看测试失败，最后写实现代码
- 每一步必须通过验证后才能进入下一步
- 连续失败 3 次必须停下来讨论架构问题

## 执行流程

### 第一步：红（写失败的测试）
1. 分析目标功能，确定测试用例
2. 编写最小化的测试代码
3. 运行测试，确认测试**失败**（红色）

#### 测试失败分类处理
| 失败类型 | 原因 | 处理方式 |
|----------|------|----------|
| AssertionError | 功能未实现（预期失败） | 进入"绿"阶段 |
| SyntaxError | 测试代码语法错误 | 先修正测试本身 |
| ImportError | 导入模块不存在 | 确认项目结构后修正 |
| NameError | 变量/函数名错误 | 修正测试代码 |
| 其他异常 | 环境或配置问题 | 排查后修正 |

**2次修正原则**：如果测试本身无法在2次修正内修复，停止并请求人工介入

### 第二步：绿（写最简单的实现）
1. 编写刚好能让测试通过的最简代码
2. 不要提前做任何优化、抽象或额外功能
3. 运行测试，确认**通过**（绿色）

### 第三步：重构
1. 在测试全部通过的前提下优化代码结构
2. 消除重复、改善命名、提取公共逻辑
3. 每次重构后立即运行全部测试
4. 保持测试持续绿灯

### 第四步：提交
- 全部通过后执行 `/commit`

## 注意
- 禁止跳过测试直接写实现
- 禁止在测试失败时继续写新功能
- 如果是修复 Bug，必须先写复现测试

## 测试用例设计指导
- 使用 Given-When-Then 模板编写测试用例
- 每个测试只验证一个行为
- 优先覆盖边界条件和异常路径
- 使用测试夹具（fixtures）减少重复设置代码

## Mock/Stub 策略
- 对外部依赖（API、数据库、文件系统）使用 mock
- 仅 mock 测试目标直接依赖的接口
- 避免过度 mock 导致测试失去意义
- 对内部纯函数优先使用真实调用

## 测试金字塔
- 单元测试（70%）：验证单一函数/方法的行为
- 集成测试（20%）：验证模块间协作
- 端到端测试（10%）：验证完整用户流程

## 失败处理（回滚机制）

- **红阶段失败**：测试本身问题 → 2次修正原则；不是预期失败 → 重新分析需求
- **绿阶段失败**：实现代码问题 → 回退修改 `git checkout -- <file>`，重新编写最简实现
- **重构阶段失败**：`git reset --hard HEAD~1` 回退到上一步，重新执行重构
- **整体回滚**：连续失败 3 次后，停止编写新代码，分析架构问题
- **无人值守**：重试 ≤3 次，失败后跳过并记录到 `.claude/reports/blocked.md`
