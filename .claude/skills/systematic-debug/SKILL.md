# 系统化调试流程

## 核心原则
- 先假设，再验证，后修复
- 禁止在不理解报错原因的情况下盲目修改代码
- 同一方案连续失败 2 次必须停止并重新分析

## 执行流程

### 第一步：收集信息
1. 读取完整的错误堆栈和日志
2. 定位错误发生的文件、行号、函数
3. 记录触发条件和环境信息

### 第二步：建立假设
1. 输出假设格式：`[假设] 错误根因是...因为...`
2. 列出关键证据（堆栈、变量状态、调用链）
3. 如果证据不足，建议增加调试点或日志

### 第三步：验证假设
1. 通过添加调试点、检查变量值、追踪调用链来验证
2. 如果假设被推翻，回到第二步建立新假设
3. 连续 2 个假设被推翻，请求人工介入

### 第四步：修复
1. 在确认根因后，制定修复方案
2. 执行最小化修复
3. 验证修复：重新运行失败的测试 + 相关模块的全量测试
4. 如果修复失败，回退并重新分析

### 第五步：提交
- 修复通过后执行 `/commit`，格式：`fix(<scope>): <修复说明>`

## 熔断规则
- 同一修复方案连续失败 2 次 → 立即停止，请求人工介入
- 单文件编辑超过 5 次 → 停止，重新分析根因
