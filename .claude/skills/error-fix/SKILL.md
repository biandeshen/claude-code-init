# 自动修复协议

## 修复流程

### 1. 信息收集
获取并整理：
- 错误信息（终端输出、测试报告、日志）
- 错误类型：编译错误/运行时错误/逻辑错误/测试失败
- 发生位置：文件名、行号、函数名
- 触发条件：什么操作导致

### 2. 根因分析
**输出假设**：
```
[假设] 错误根因是 <具体原因>
依据：<证据1>、<证据2>
```

验证步骤：
- 读取相关代码
- 检查 git 历史（最近谁改了什么）
- 如不确定，增加调试点

### 3. 方案对比
提出至少两种方案：

| 方案 | 修改范围 | 风险 | 推荐 |
|------|----------|------|:----:|
| 最小修复 | 只改一行 | 低 | ✅ |
| 根本解决 | 重构相关逻辑 | 中 | |

选择最佳方案并说明理由。

### 4. 执行与验证
1. 应用修改
2. 运行失败的测试
3. 运行相关模块测试
4. 检查是否引入新问题

**失败处理**：
- 回退修改
- 分析失败原因
- 同一方案不得连续尝试超过 2 次

### 5. 提交
```bash
git commit -m "fix(<scope>): <修复说明>"
```

## 熔断机制

| 触发条件 | 动作 |
|----------|------|
| 同一修复方案失败 2 次 | 停止，输出分析报告，请求人工介入 |
| 单文件编辑超过 5 次 | 停止，重新根因分析 |
| 无法确定根因 | 停止，请求人工介入 |

## 输出格式

```markdown
### 错误分析
- 类型：<编译/运行时/逻辑/测试>
- 位置：<file:line>
- 触发：<如何触发>

### 根因假设
[假设] <具体原因>

### 方案对比
| 方案 | 范围 | 风险 | 推荐 |
|------|------|------|:----:|
| ... | ... | ... | ✅ |

### 执行结果
- 修改文件：<list>
- 测试结果：<通过/失败>
- 提交：<commit>
```

## 安全规则

- 不修改未涉及错误的代码
- 不在修复时引入新功能
- 保持测试覆盖率不下降
