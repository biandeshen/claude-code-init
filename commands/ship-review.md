# /ship-review - 发布前全面审查

> 多 Agent 并行审查 + 交叉验证 + 发布决策，一键完成发布前质量评估。

## 概述

`/ship-review` 是发布前的最终质量门禁。它启动 4 个子 Agent 从不同维度并行审查代码库，然后交叉验证发现，最终给出 SHIP / NO SHIP 决策。

## 执行模式

### 双重执行模式

| 模式 | 条件 | 说明 |
|------|------|------|
| **并行模式**（优先） | Claude Code sub-agent 可用 | 启动 4 个子 Agent 并行审查，汇总结果 |
| **顺序模式**（降级） | sub-agent 不可用 | 当前会话内顺序执行 4 个角色 |

> 检测 sub-agent 可用性：通过 `/team` 命令或 Claude Code 内置 agent 机制判断。如果无法确认，默认使用顺序模式。

## Phase 1 — 子 Agent 并行审查

启动 4 个专业子 Agent 同时审查：

### Agent A: 安全与代码质量
```
角色：代码审查专家
任务：
  1. 加载 code-review skill 进行 OWASP Top 10 全覆盖安全检查
  2. 审查 git diff（当前分支 vs main/master）
  3. 检查代码质量：函数长度、命名、重复代码
  4. 输出：安全漏洞清单 + 代码质量问题清单（严重度分级）
```

### Agent B: 架构一致性
```
角色：架构审查专家
任务：
  1. 对照 CLAUDE.md 检查变更是否符合项目架构规则
  2. 检查模块间耦合是否合理
  3. 检查 API/DB schema 变更是否有破坏性影响
  4. 输出：架构问题清单 + 耦合风险矩阵
```

### Agent C: 自动化校验
```
角色：质量门禁执行者
任务：
  1. 运行所有 check_*.py 脚本（secrets, function_length, dependencies, import_order, project_structure, docs_consistency, trigger_conflicts）
  2. 运行 check_ecc.sh 和 validate_skills.sh
  3. 运行 pre-commit run --all-files
  4. 输出：校验结果汇总（通过/失败/警告）
```

### Agent D: 文档完整性
```
角色：文档审查专家
任务：
  1. CHANGELOG.md 是否记录了本次变更？
  2. GUIDE.md 和 help.md 的命令引用是否仍有效？
  3. CLAUDE.md 的项目结构图是否仍然准确？
  4. 是否存在过时的文件路径引用？
  5. 模板版本头是否需要 bump？
  6. Router 触发词表是否与新/改动的命令一致？
  7. 输出：文档问题清单
```

## Phase 2 — 交叉验证

**由主会话执行**，综合 4 个子 Agent 的所有发现：

### 2.1 构建发现矩阵

将 4 个 Agent 的所有发现汇总为统一矩阵：

| ID | 来源 | 维度 | 发现 | 严重度 | 文件/位置 |
|----|------|------|------|:------:|-----------|
| F1 | Agent A | 安全 | SQL 注入风险 | 🔴 严重 | api/users.js:42 |
| F2 | Agent B | 架构 | 循环依赖 | 🟡 中 | src/modules/ |
| ... | ... | ... | ... | ... | ... |

### 2.2 矛盾检测

对比不同 Agent 的发现，检测矛盾或遗漏：

- Agent A 发现安全漏洞但 Agent B 未提及相关耦合问题？
- Agent C 校验通过但 Agent A 报告了相关问题？
- Agent B 标记架构问题但 Agent D 未在文档中反映？

### 2.3 盲点扫描

主动扫描 4 个 Agent 都未覆盖的维度：

| 盲点维度 | 检查项 |
|----------|--------|
| 性能 | 是否有 N+1 查询？是否有不必要的重渲染？ |
| 向后兼容 | API 变更是否破坏已有客户端？ |
| i18n | 新增文案是否可翻译？ |
| 可访问性 | UI 变更是否符合 a11y 标准？ |
| 日志与监控 | 关键路径是否有日志？ |

### 2.4 综合评分

对每个发现进行 impact × likelihood 评估：

```
综合评分 = 影响范围 (1-5) × 发生概率 (1-5)

影响范围:
  5 = 全系统宕机/数据丢失
  4 = 核心功能不可用
  3 = 部分功能受影响
  2 = 边缘情况
  1 = 仅视觉/文案

发生概率:
  5 = 必然触发
  4 = 高概率场景
  3 = 常见场景
  2 = 特定条件
  1 = 极端边缘
```

## Phase 3 — 发布决策

### 3.1 问题分级

| 级别 | 条件 | 含义 |
|:----:|------|------|
| 🔴 阻断项 | 严重安全漏洞 / 数据丢失风险 / 综合评分 ≥ 20 | **必须修复**才能发布 |
| 🟡 建议修复 | 高风险但非阻断 / 综合评分 12-19 | **强烈建议**，但非强制 |
| 🟢 备注 | 低风险 / 综合评分 < 12 / 文档类 | 记录即可，不影响发布 |

### 3.2 最终裁决

基于阻断项数量做出决策：

| 裁决 | 条件 |
|:----:|------|
| ✅ **SHIP** | 0 个阻断项，≤ 3 个建议修复项 |
| ⚠️ **SHIP WITH CAVEATS** | 0 个阻断项，4+ 个建议修复项（附风险说明） |
| ❌ **NO SHIP** | ≥ 1 个阻断项 |

### 3.3 输出格式

```markdown
# 发布审查报告 — {日期}

## 审查摘要
- 变更文件数：{N}
- 子 Agent 数：4（并行）
- 总发现问题数：{N}（🔴 {n} / 🟡 {n} / 🟢 {n}）
- 盲点发现：{N}

## 🔴 阻断项（{N} 项）
| 发现 | 来源 | 文件 | 评分 | 修复建议 |
|------|------|------|:----:|----------|
| ... | ... | ... | 25 | ... |

## 🟡 建议修复（{N} 项）
| 发现 | 来源 | 文件 | 评分 | 建议 |
|------|------|------|:----:|------|
| ... | ... | ... | 15 | ... |

## 🟢 备注（{N} 项）
| 发现 | 来源 | 文件 | 说明 |
|------|------|------|------|
| ... | ... | ... | ... |

## 交叉验证发现
- 矛盾：{描述}
- 盲点：{描述}

## 🏁 最终裁决
**{SHIP / SHIP WITH CAVEATS / NO SHIP}**

{理由}

---
*审查完成于 {datetime} | /ship-review*
```

## 使用示例

```
/ship-review                    # 默认：审查当前分支 vs main 的全部变更
/ship-review --target main      # 指定对比目标分支
/ship-review --scope HEAD~5     # 限定审查范围（最近 5 个提交）
```

## 注意事项

- **首次运行**可能需要 3-5 分钟（4 个子 Agent 并行）
- 如果 sub-agent 机制不可用，自动降级为顺序模式（约 8-12 分钟）
- 阻断项 ≥ 1 时，必须先修复再重新审查
- 审查结果自动保存到 `.claude/reports/ship-review-{date}.md`

## 与其他命令的关系

| 命令 | 关系 |
|------|------|
| `/review` | Ship-review 的一部分（Agent A 的输入） |
| `/validate` | Ship-review 的一部分（Agent C 的输入） |
| `/architect` | Ship-review 的一部分（Agent B 的输入） |
| `/team` | Ship-review 使用相同的 Agent Teams 机制 |

`/ship-review` 是 `/review` + `/validate` + `/architect` + 文档审查的超集，附加交叉验证和发布决策。
