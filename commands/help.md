# /help - 智能帮助系统

根据用户输入，提供场景导航、完整命令清单或单个命令详解。

## 执行逻辑

### 步骤 0：上下文感知（优先执行）

分析当前对话历史，判断用户当前最可能需要的场景：

| 当前上下文 | 优先推荐 |
|------------|----------|
| 刚完成代码编写 | `/review` — 代码审查 |
| 刚修复了一个 bug | `/commit` — 规范提交 |
| 正在讨论技术方案 | `/architect <方案>` — 架构评审 |
| 刚完成多个功能 | `/team 3` — 启动多人审查 |
| 快到下班时间 | `bash scripts/tmux-session.sh` — 启动过夜任务 |

如果匹配到上述场景，**首先输出推荐的 1-2 个命令**，再输出完整的场景导航表格。

| 用户输入 | 执行动作 |
|----------|----------|
| `/help` | 显示场景导航表格 |
| `/help commands` 或 `/help 列表` | 显示完整命令清单 |
| `/help <command>` | 查看该命令的详细说明（如 `/help review`） |
| `/<command>` | 直接执行该命令（如 `/review`） |
| 其他组合 | 提示正确用法 |

> 💡 `/help <命令>` 查看命令详情，`/<命令>` 直接执行。例如：`/help review` 查看审查命令怎么用，`/review` 直接开始审查。

## 一、场景导航（无参数）

## 你想做什么？

| 场景 | 使用命令 | 简要说明 |
|------|----------|----------|
| 📋 检查代码质量 | `/review` | 审查当前分支相对于 main 的所有改动 |
| 📦 提交代码 | `/commit` | 自动分析修改、生成规范提交信息、运行 pre-commit |
| 🏗️ 评审技术方案 | `/architect <方案>` | 多维度架构分析与优先级建议 |
| 🔧 修复错误 | `/fix` | 系统化分析 → 根因 → 方案 → 验证 → 提交 |
| 🧹 重构代码 | `/refactor` | 安全重构：先写保护测试，逐步执行，保持测试绿灯 |
| 📖 理解代码 | `/explain <文件/函数>` | 深度解释架构、数据流、关键决策和潜在风险 |
| ✅ 运行项目校验 | `/validate` | 运行结构、密钥、函数长度、依赖、导入五项检查 |
| 🧠 管理项目记忆 | `/remember` | 浏览/搜索/编辑/清理项目记忆（架构决策/Bug根因/编码模式） |
| 🧩 开发复杂功能 | `/opsx:propose <名>` | 启动 SDD 流程：提案→规格→设计→任务→检查→执行 |
| 💡 不知道用什么 | 直接描述你想做什么 | 我会推荐合适的命令 |

> 💡 输入 `/help commands` 查看所有可用命令的完整清单
> 💡 输入 `/help <命令>` 查看某个命令的详细说明
> 💡 关于项目整体规范，可查阅 README.md、GUIDE.md、templates/SOUL_Template.md

### Superpowers 核心能力
Superpowers 通过自然语言触发，无需记忆命令：
- 说 `用 TDD 方式实现这个函数` → 强制红/绿/重构循环
- 说 `子代理驱动开发` → 自动并行执行子任务
- 说 `帮我找到这个 bug 的根因` → 触发根因追踪模式

## 三、单个命令帮助（`/help <command>` 触发）

如果用户输入 `/help review`，读取并输出 `commands/review.md` 的内容摘要。
如果用户输入 `/help commit`，读取并输出 `commands/commit.md` 的内容摘要。
以此类推。

## 重要规则
- 保持简介，不主动列出所有命令，除非用户明确要求 `commands`
- 对不明确的输入，引导用户重新输入或给出示例
