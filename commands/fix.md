# /fix - 错误自动修复

对当前项目的错误进行系统化修复。

## 修复流程

### 1. 信息收集
- 分析所有最近的错误日志（终端输出、测试失败报告、会话历史）
- 收集：错误类型、发生位置、频率、触发条件

### 2. 根因分析
- 输出假设：`[假设] 错误根因是...因为...`
- 列出关键证据（错误堆栈、相关代码行、环境变量）
- 在验证假设前，不得修改任何代码

### 3. 修复方案对比
- 提出至少两种方案：最小侵入 vs 根本解决
- 评估维度：修改范围、风险、是否引入新依赖、对现有功能的影响
- 选择最佳方案并说明理由

### 4. 执行与验证
- 执行选择的修复方案
- 验证：重新运行失败的测试 → 运行关联模块的全量测试 → 检查是否引入新错误
- 失败处理：回退，不得连续尝试同一方案超过 2 次；超过 2 次立即请求人工介入

### 5. 提交
- 修复通过全部测试后，自动执行 `/commit`
- Commit 格式：`fix(<scope>): <修复说明>`

## 熔断机制
- 同一修复方案连续失败 **2 次** → 停止，请求人工介入
- 单文件编辑超过 **5 次** → 停止，重新分析根因

## 输出格式
以 Markdown 输出：

### 错误分析
- 错误类型
- 位置
- 触发条件

### 根因假设
`[假设] ...`

### 修复方案对比
| 方案 | 范围 | 风险 | 推荐 |
|------|------|------|:---:|
| 方案A | ... | ... | ✅ |
| 方案B | ... | ... | |

### 执行结果
- 修复的文件清单
- 测试结果（通过/失败数量）
- 提交 commit 信息
