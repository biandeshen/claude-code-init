# /fix - 系统化错误修复

对当前项目的错误进行系统化修复，遵循分析→根因→方案→验证→提交的闭环流程。

## 修复流程

### 1. 信息收集
- 获取最近的错误信息（终端输出、测试报告、运行日志）
- 明确：错误类型、发生位置、频率、触发条件

### 2. 根因分析
- 输出假设：`[假设] 错误根因是...因为...`
- 列出关键证据（堆栈、代码行、相关变量状态）
- 在验证假设前 **不得修改代码**；如不确定，增加调试点

### 3. 方案对比
- 提出至少两种方案（最小修复 vs 根本解决）
- 评估维度：修改范围、风险、是否引入新依赖、对其他模块的影响
- 选择最佳方案并说明理由

### 4. 执行与验证
- 执行选择的修复
- 验证步骤：
  1. 重新运行失败的测试
  2. 运行修复模块相关的所有测试
  3. 检查是否引入新错误或警告
- 若失败，回退修改，分析原因，**不允许同一方案连续尝试超过 2 次**

### 5. 提交
- 修复通过所有测试，自动调用 `/commit` 生成规范提交
- 提交格式：`fix(<scope>): <修复说明>`

## 熔断规则
- 同一修复方案连续失败 **2 次** → 立即停止，输出分析报告，请求人工介入
- 单文件累计编辑 **5 次** → 强制停止，重新进行根因分析，必要时提出更多调试信息

## 输出格式

### 错误分析
- 类型
- 位置
- 触发条件

### 根因假设
`[假设] ...`

### 修复方案对比
| 方案 | 修改范围 | 风险 | 推荐 |
|------|----------|------|:---:|
| 方案A | ... | ... | ✅ |
| 方案B | ... | ... | |

### 执行结果
- 修改文件清单
- 测试结果（通过数/总数）
- 提交信息
